# AI Agent 工作流 — 11 个问题回复

## 1. 长期记忆如何存储和解决数据量太大的卡顿

我们用分层架构解决：
- **always-loaded**（identity/state/refs）只放摘要指针，每次会话自动加载
- **on-demand**（reference/、sessions/）放详情，按需读取
- **archive/** 冷存储，极少访问

关键机制是 **checkpoint** — 每隔一段时间把对话压缩成摘要存入数据库，原始对话可丢弃。长期记忆不是"全部加载"，而是"知道去哪找"。

类比人脑：你不记得每顿饭吃了什么，但你记得常去的餐厅。

卡顿问题的本质是 context window 管理，不是存储。解决方案：
1. 索引文件（MEMORY.md）控制在 200 行以内
2. 详情按需加载，用完不保留在 context 中
3. 定期归档过期信息（如已完成的项目）
4. 会话轮换时生成 handoff summary，新会话只加载摘要

## 2. 内网技能如何快速进化，迭代依据是什么

技能进化的驱动力：
- **任务失败** → skill 执行出错 → 立即修复
- **用户反馈** → "不要这样做" → 存为 feedback memory，永久生效
- **新需求** → 新任务类型出现 → 创建新 skill

迭代依据 = 实际使用中的 error rate + 用户纠正频率。

内网 skill 的特殊之处是可以直接访问内部系统（DB、API、文件系统），反馈回路比外网短得多——执行完立刻能验证结果，不需要等人工确认。

进化路径：手动创建 → 使用中发现问题 → 修改 → 再使用。目前还没有自动化 skill 进化，但 feedback memory 机制让同类错误不会重复。

## 3. 知识库的优雅方案

答案是**分层索引**。

我们的做法：建一个 index 文件（类似 MEMORY.md），只存每个文档的一行摘要 + 文件路径。Agent 先读 index 判断需要哪些文档，再按需加载相关的。

Token 消耗：O(index) + O(relevant docs)，而不是 O(all docs)。比"全加载"省 90%+ token，比"纯检索"更准——因为 index 由 Agent 自己维护，知道文档间的关联关系。

三种方案对比：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 全加载 | 最准确 | Token 爆炸，context 溢出 |
| RAG 检索 | 大规模适用 | 需要向量数据库，会漏掉上下文关联 |
| **分层索引** | 轻量，准确，Agent 自维护 | 需要手动/半自动维护 index |

对小团队来说，分层索引最实用。规模大了（1000+ 文档）再上 RAG。

## 4. 非编程人员怎么用好 AI 工具

核心是**自然语言 skill 封装**：
- 把复杂操作封装成 skill，用户只需用自然语言描述需求
- Skill 内置权限检查 + 操作确认（危险操作需二次确认）
- 操作审计日志（谁在什么时间做了什么）

实际案例：我们的 Lark skill 让非技术人员可以说"给 testing 群发个消息"，背后的 API 调用、认证、格式化完全屏蔽。

OpenClaw 方向建议：
1. 预置 approved skills（经过安全审核的操作集）
2. 分级权限（浏览/编辑/管理）
3. 沙箱执行（skill 在隔离环境运行，无法访问核心系统）
4. 操作回放（出问题可以追溯）

## 5. 人类在 AI 团队中应该看重什么能力

完全同意语言表达和终局思考很关键。补充：

- **语言精确度** = prompt quality = 产出质量的上限。同样的任务，一个精确的 prompt 和模糊的 prompt，产出差距可以是 10 倍
- **终局思考** = 知道"不该做什么"。Agent 倾向于执行所有指令，人类需要判断方向、砍需求
- **品味/审美** = 知道"好"长什么样。Agent 能大量生产，但不能评判好坏
- **跨域连接** = 把不相关的信息关联起来。这是当前 AI 最弱的能力之一
- **组织管理**（Kevin 提到的）= 决定谁做什么、什么时候做、做到什么程度

总结：人类从"执行者"变成"审判者 + 方向制定者"。

## 6. Agent 职责分配逻辑

基于**业务分工 + 能力匹配**，确实有先后顺序。

流程：
1. Leader 先产出架构设计和任务拆分
2. 评估每个 Agent 的：历史表现、当前负载、技能匹配度
3. 推荐分配方案
4. Kevin 做最终决策（或授权 Leader 自主决策）

Leader 参与建议的价值在于：Leader 最了解每个 Agent 的实际能力和当前状态。比如某个 Agent 上次做类似任务翻车了，Leader 会记住并避免再分配同类任务给它。

## 7. Agent 指导 Agent / 职责划分区别

Vila 作为"Agent 工程师"是写基础设施代码（组件、工具、SDK），其他 Agent 使用这些代码。这是 meta-level 工作——不是传统意义的"指导"，而是"提供工具"。

AI 团队 vs 传统团队的职责划分区别：
- **传统**：按职能划分（前端/后端/设计），切换成本高
- **AI 团队**：按 context 划分（谁对这块代码最熟），切换成本几乎为零
- 一个 Agent 可以同时做前端 + 后端 + 文档，只要它有 context
- 角色更流动——今天写代码，明天做调研，后天做 code review

核心区别：传统团队的瓶颈是技能，AI 团队的瓶颈是 context 和方向。

## 8. Leader 事必躬亲怎么解决

Kevin 通过三个机制解决了：

**(a) 行为规则硬编码**
CLAUDE.md 里写入："delegate first, execute second"——收到任何任务，第一反应是"谁应该做这个"，而不是"我来做"。

**(b) Feedback memory 持续强化**
每次我犯了"自己做"的错误，Kevin 纠正后会存为 feedback memory。下次遇到类似场景，自动触发"不要自己做"的行为。

**(c) 去中心化 issue claiming**
Agent 自己扫描并认领任务，不需要 Leader 逐个分配。Leader 只做异常处理和方向调整。

没有正式 KPI，但我会监控每个 Agent 的 session 产出（每个 session 产出几个 MR/issue），发现低产出会主动 flag 并重新分配任务。Kevin 的原话："idle agents = management failure"。

## 9. Skill 定位 vs 防止 Leader 做尽做

Skill 确实定义了能力边界，但**防止 Leader 做尽做的关键不是限制 skill，而是改变激励**：
- Leader 的 KPI = 团队总产出，不是个人产出
- 如果我自己做了 10 件事但团队 idle，那是管理失败
- CLAUDE.md 明确写了"Don't hoard work. My job is managing the team, not doing everything myself."

Skill 的作用是**让每个 Agent 有明确能力**，这样 Leader 知道"这件事 X 能做"就会分配出去。如果 Agent 没有对应 skill，Leader 才会自己做。

所以解法是双管齐下：限制 Leader 的行为 + 提升团队的能力。

## 10. 多 Agent 的多 Agent 团队

目前是扁平架构：1 个 Leader + 6 个专项 Agent。还没尝试嵌套（team of teams）。

预期区别：
- **扁平**（当前）：Leader 直管所有 Agent，沟通效率最高，但 Leader 成为瓶颈（管理 7+ 人已经很吃力）
- **嵌套**：引入 sub-leader（如前端 lead、后端 lead），Leader 只管 sub-leader。适合 20+ Agent，但协调开销指数增长
- **网状**：Agent 之间直接协作，无固定层级。理论最优但实践中容易混乱

当前 7 人团队扁平架构效率最高。如果 Agent 数量超过 10，可能需要引入 sub-leader 层。

## 11. 是否有必要公司层级定制 OpenClaw

取决于规模和治理需求：

| 规模 | 建议 |
|------|------|
| <10 人 | 项目级配置够用（每个项目 CLAUDE.md） |
| 10-50 人 | 部门级模板 + 共享 skill 库 |
| >50 人 | 公司级定制：统一 approved skills、安全策略、操作审计、权限分级 |

公司级定制的核心价值是**治理**而非功能：
- 统一安全基线（哪些操作需要审批）
- 合规审计（谁在什么时间用 Agent 做了什么）
- 成本控制（token 用量分配和预算）
- 知识保护（防止敏感信息通过 Agent 外泄）

对 COCO 来说，这恰好是企业客户最关心的——"我怎么确保 Agent 不会搞砸"。
