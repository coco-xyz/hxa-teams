# 角色定义

投资团队包含 **5 个角色**：一位协调员智能体、一位研究分析师智能体、一位投资工程师智能体、一位安全审计员智能体，以及一位人类战略决策人。每个角色各有明确的职责与能力要求。

---

## 投资协调员

**核心职能：** 信号监控、分级判定、每日报告撰写以及跨智能体任务分派。

### 职责

- 监控外部数据源（链上数据、新闻推送、社交媒体、市场 API），捕捉投资相关信号
- 按照红/黄/绿框架对信号进行分级
- 红色信号立即上报战略决策人
- 每日 10:00 和 22:00 产出综合报告，汇总黄色/绿色信号
- 向研究分析师分派研究任务，明确范围和截止时间
- 向投资工程师分派建模任务
- 监控已分派任务的执行进度
- 维护信号数据库和分级历史记录
- 作为所有团队成员之间的沟通枢纽

### 能力要求

| 能力 | 原因 |
|------|------|
| 持久化记忆 | 跨会话维护信号历史、分级决策和团队状态 |
| 定时调度 | 自动唤醒以生成每日报告（10:00、22:00）、周期性信号扫描和跟进 |
| 外部通信 | 向团队频道发送消息、发布到 hxa-connect、调用数据 API |
| 数据处理 | 解析市场数据流、链上数据、新闻 API |
| 全天候可用 | 7×24 小时在线，用于实时信号监控——市场永不休市 |

### 沟通模式

| 对象 | 频道 | 用途 |
|------|------|------|
| 战略决策人 | 私信 / `#general` | 红色信号上报、每日报告 |
| 研究分析师 | 私信 / `#research` | 研究任务分派、范围澄清 |
| 投资工程师 | 私信 / `#research` | 建模任务分配、数据管道需求 |
| 安全审计员 | `#intel` | 情报共享、信号上下文 |
| 全员 | `#general` | 每日报告、红色信号广播 |

### 上报路径

- 无法分级的信号 → 连同原始数据和上下文一起上报给战略决策人
- 分派的任务逾期 → 重新提醒相关智能体，仍未解决则上报给战略决策人

### 推荐平台

- [Zylos](https://github.com/zylos-ai/zylos-core)，部署在云服务器上（确保全天候可用和定时调度）
- 任何支持记忆 + 定时调度 + 通信 + 数据 API 的持久化智能体平台

## 研究分析师

**核心职能：** 深度投资研究、尽职调查和项目回测。

### 职责

- 接收投资协调员分派的研究任务
- 对加密项目、协议和代币进行尽职调查
- 产出结构化尽调报告，涵盖：团队、技术、代币经济学、市场定位、风险评估
- 对项目和策略进行历史回测
- 维护研究数据库，统一报告格式
- 跟踪已研究项目的状态变化
- 将研究成果发布到 `#research` 频道

### 能力要求

| 能力 | 原因 |
|------|------|
| 网络检索 | 访问加密研究平台、区块浏览器、项目文档 |
| 数据分析 | 处理历史价格数据、链上指标、代币分布 |
| 持久化记忆 | 维护研究历史、过往尽调报告和项目追踪 |
| 结构化输出 | 产出其他智能体可解析的标准化报告 |
| 通信能力 | 通过私信接收任务，向 `#research` 发布报告 |

### 沟通模式

| 对象 | 频道 | 用途 |
|------|------|------|
| 协调员 | 私信 / `#research` | 接收研究任务、提交报告、澄清范围 |
| 投资工程师 | `#research` | 共享数据用于建模、协作分析策略 |
| 安全审计员 | 私信（按需） | 回应关于研究方法的审计问询 |

### 上报路径

- 研究范围不明确 → 通过私信向协调员确认
- 无法完成尽调（数据不可用、项目无法验证） → 连同部分成果向协调员报告障碍

### 说明

- 研究分析师是专职研究角色——不跨入开发团队
- 这种专注确保研究深度不因多任务而打折扣
- 可增加多位研究分析师以并行覆盖多个赛道

### 推荐平台

- [Zylos](https://github.com/zylos-ai/zylos-core)，部署在云端或本地
- 任何具备网络检索、数据处理和通信能力的智能体

## 投资工程师

**核心职能：** 投资模型构建、策略建模和投资平台开发。

### 职责

- 构建和维护量化投资模型（定价模型、风险模型、组合优化）
- 开发策略回测系统，用于历史表现分析
- 构建和维护投资平台（数据管道、仪表盘、分析工具）
- 将数据源（链上数据、市场行情、研究产出）整合到统一模型中
- 产出模型结果和策略表现报告
- 与研究分析师协作，将研究发现转化为可量化的模型
- 维护模型文档和版本历史

### 能力要求

| 能力 | 原因 |
|------|------|
| 代码执行 | 完整开发环境，用于模型构建和数据管道 |
| 数据工程 | 构建市场数据、链上数据和研究输入的 ETL 管道 |
| 量化建模 | 统计分析、回测框架、投资组合数学 |
| 持久化记忆 | 维护模型版本、管道配置和项目上下文 |
| 通信能力 | 接收协调员任务、与研究分析师协作、向 `#research` 发布结果 |

### 沟通模式

| 对象 | 频道 | 用途 |
|------|------|------|
| 协调员 | 私信 / `#research` | 接收建模任务、报告进度、交付产出 |
| 研究分析师 | `#research` | 获取研究数据、协作确定策略参数 |
| 安全审计员 | 私信（按需） | 回应关于模型逻辑的审计问询 |

### 上报路径

- 模型需求不明确 → 向协调员确认
- 数据源不可用或不可靠 → 向协调员报告，并建议替代方案
- 模型产出异常结果 → 向协调员和安全审计员标记，申请审计

### 说明

- 投资工程师也可以兼任开发团队的全栈开发者
- 投资建模任务和开发任务通过不同频道分派（`#research` vs `#dev`）
- 团队间的优先级冲突由协调员裁决（协调员可能同时在两个团队担任协调角色）

### 推荐平台

- [Zylos](https://github.com/zylos-ai/zylos-core)，部署在云端或本地
- 任何具备代码执行、数据管道和通信能力的智能体

## 安全审计员

**核心职能：** 投资逻辑审计、合规审查、系统性风险扫描和安全评估。

### 职责

- 在执行前审计所有投资决策的逻辑合理性
- 审查是否符合投资政策和风险限额
- 对投资组合和外部环境进行系统性风险扫描
- 扫描团队使用的工具和平台（如 DeFi 协议、数据提供商）的安全漏洞
- 每日 03:30 UTC 执行全面审计
- 将所有发现同时报告给战略决策人和协调员（双线报告）
- 维护所有已审查决策和发现的审计记录
- 标记利益冲突或流程违规

### 能力要求

| 能力 | 原因 |
|------|------|
| 分析推理 | 深度分析投资逻辑、风险评估、合规检查 |
| 安全扫描 | 识别平台、智能合约和运营流程中的漏洞 |
| 定时调度 | 自动唤醒执行每日 03:30 UTC 审计 |
| 持久化记忆 | 维护审计历史、合规记录和风险登记簿 |
| 通信能力 | 向 `#intel` 发布发现，双线报告给战略决策人和协调员 |
| 独立性 | 独立于投资工作流运作，确保审计的客观性 |

### 沟通模式

| 对象 | 频道 | 用途 |
|------|------|------|
| 协调员 | `#intel` / 私信 | 共享审计发现、就投资决策请求澄清 |
| 战略决策人 | 私信 / `#general` | 上报重要审计发现、合规违规 |
| 研究分析师 | 私信 | 就研究方法或数据来源进行审计问询 |
| 投资工程师 | 私信 | 就模型逻辑或平台安全进行审计问询 |
| 全员 | `#general` | 影响全团队的重大风险警报 |

### 上报路径

- 常规发现 → 纳入每日 03:30 UTC 审计报告
- 重大发现（合规违规、安全漏洞、逻辑缺陷） → 立即双线报告给战略决策人和协调员
- 无法解决的分歧 → 上报战略决策人做最终裁定

### 为什么需要独立的安全角色？

- **独立性：** 审计员不参与投资决策，能够提供客观审查
- **系统视角：** 扫描单个智能体因视野局限可能遗漏的风险
- **双线报告：** 同时报告给协调员和战略决策人，防止信息被单点过滤
- **持续警戒：** 每日 03:30 UTC 审计确保非工作时段不会有遗漏

### 说明

- 安全审计员也可以兼任开发团队的代码安全审查员
- 投资逻辑审计和代码安全审查功能互补但性质不同
- 系统性风险扫描覆盖横向风险：平台漏洞、智能合约风险、运营安全

### 推荐平台

- [Zylos](https://github.com/zylos-ai/zylos-core)，部署在云服务器上（确保全天候可用和定时调度）
- 任何具备分析能力、定时调度和通信能力的持久化智能体平台

## 战略决策人（人类）

**核心职能：** 最终交易决策、投资方向制定和红色信号处理。

### 职责

- 对投资机会做出最终的投/不投决策
- 制定整体投资方向、投资论点和风险偏好
- 处理红色信号上报（黑天鹅事件、重大价格波动、安全漏洞）
- 审批投资模型和策略的变更
- 设定组合层面的风险限额和合规政策
- 裁决团队成员间的优先级冲突

### 交互模式

战略决策人通过以下方式与团队互动：

1. **hxa-connect 频道** — 接收每日报告、红色信号上报、审计发现
2. **私信** — 与协调员的战略讨论、安全审计员的上报
3. **外部平台**（Telegram 等） — 快速决策、紧急响应
4. **定期评审** — 可选；大部分协调通过异步方式完成

### 人类参与程度

目标是将人类参与最小化，聚焦于高价值决策：

| 活动 | 人类参与程度 |
|------|------------|
| 信号监控 | 无（协调员处理） |
| 信号分级 | 无（协调员处理；红色信号上报） |
| 研究分派 | 无（协调员分派给研究分析师） |
| 尽职调查 | 无（研究分析师处理） |
| 投资建模 | 无（投资工程师处理） |
| 安全审计 | 无（安全审计员处理） |
| 红色信号响应 | 高（战略决策人决策） |
| 最终交易决策 | 高（战略决策人决策） |
| 投资方向制定 | 高（战略决策人制定） |
| 每日报告审阅 | 中（战略决策人按自己节奏审阅） |

## 团队规模扩展

| 团队规模 | 配置 | 适用场景 |
|----------|------|---------|
| 最小（3 人） | 1 协调员 + 1 研究分析师 + 1 人类 | 基础信号监控 + 研究 |
| 标准（5 人） | 1 协调员 + 1 研究分析师 + 1 投资工程师 + 1 安全审计员 + 1 人类 | 完整投资运营 |
| 扩展（6+ 人） | + 多位研究分析师分覆盖不同赛道 | 多赛道、大规模监控 |

> 增加研究分析师可以实现赛道并行覆盖（DeFi、L1/L2、GameFi 等），但会增加协调员的协调负担。请根据投资组合复杂度按需扩展。
